# Hytale Lighting Export Analysis

## Scope

This document captures the current investigation into why blocks generated by WorldPainter appear to have different lighting from blocks placed manually inside Hytale.

## Current Findings

### 1. Exported chunk lighting is effectively sky-only

The current Hytale export path does not compute or preserve emitted block light in a meaningful way.

- `HytaleWorldExporter.HytaleRegionMinecraftWorld#getBlockLightLevel(...)` returns `0`
- `HytaleWorldExporter.HytaleRegionMinecraftWorld#getSkyLightLevel(...)` returns `15`
- the corresponding light setters are no-ops

This means the exporter-side world abstraction does not provide real per-block lighting state to later stages.

### 2. Serializer writes calculated sky light into both light channels

`HytaleBsonChunkSerializer` repeatedly calls `writeCalculatedSkyLightData(...)` when writing light payloads.

Observed behavior during investigation:

- local light and global light are both populated from the same simplified skylight path
- mixed sections fall back to writing full skylight data for the whole section
- no completed emitted-light propagation path was found in the active export flow

This is consistent with worlds where terrain is visible, but emissive blocks do not match in-game placed blocks.

### 3. Block emission metadata is not populated

`HytaleBlockRegistry` currently initializes `lightEmission = 0` for loaded block definitions in the code paths inspected.

`HytaleBlock` itself also carries no intrinsic lighting metadata, so the export pipeline has no reliable emitter value to propagate even before serialization.

### 4. The data model has room for light, but the exporter is not feeding it correctly

`HytaleChunk` stores both block light and sky light arrays, and `HytaleChunkLightDataBuilder` exists as a lower-level helper for Hytale light storage. The issue is not that the format has nowhere to store light; the issue is that the current export pipeline does not appear to compute and serialize emitted light correctly.

## Root Cause Statement

The lighting mismatch occurs because the current Hytale export path does not carry real emitted-light metadata from block definitions through chunk generation and into BSON serialization, and instead writes a simplified skylight representation into the serialized light data.

## What This Explains

- torches, lamps, lava, and other emissive blocks generated by WorldPainter can look wrong compared to blocks placed manually in Hytale
- lighting may look uniformly bright, flattened, or otherwise inconsistent after export
- differences are likely largest in enclosed or underground areas where emitted light matters more than skylight

## What Still Needs Verification Before Implementing a Fix

1. Confirm the exact Hytale meaning of the local and global RGBS light payloads for block sections.
2. Verify whether Hytale expects propagated emitted light to be precomputed in the save data, or whether some portion is recomputed on load.
3. Build a trustworthy emitter lookup from Hytale block definitions or asset metadata.
4. Decide whether to integrate emitted-light generation through `HytaleChunkLightDataBuilder`, a new propagation pass, or both.
5. Validate whether the export should write full RGB light values or whether scalar emission is expanded by the engine.

## Proposed Fix Order

1. Populate reliable `lightEmission` metadata for Hytale block definitions.
2. During export, assign block-emitter values into `HytaleChunk` for emissive blocks.
3. Add or reuse a propagation pass so emitted light spreads through sections before serialization.
4. Replace the serializer's sky-only fallback path with proper emitted-light plus skylight output.
5. Add regression coverage with at least one emissive block test and one underground enclosed-light test.

## Risk Notes

- This is not a safe quick fix. The light format is more complex than the chunk-coordinate issue and needs verification against actual Hytale expectations.
- A partial fix that only changes serializer bytes without a correct propagation model is likely to produce broken or unstable lighting.
- If Hytale re-lights chunks on load in some situations, the export fix may still need to match engine expectations closely to avoid visible popping or delayed correction.
